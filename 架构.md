# 适配器架构

## 适配器结构图

```
【App 层】 (业务逻辑：我要画个红心)
   │
   ▼ 调用 Wrapper_DrawPixel()
【Wrapper 层】 (统一菜单：谁注册了我，我就调谁的画点函数)
   ▲ 
   │ (2. 向上注册：把 Driver 的画图技能给 Wrapper)
   │
【Porting 层】 (装配车间：唯一知道全貌的地方) 
   │
   │ (1. 向下注册：把 MCU 的 SPI/GPIO 给 Driver)
   ▼
【Driver 层】 (具体外设：ST7789，使用注册来的 SPI 发送 0x2A 指令)
   │
   ▼ 调用 io->write_spi() 实际上执行了 My_SPI_Write()
【MCU HAL库】 (单片机底层：STM32/ESP32 的具体寄存器操作)
```

## 方法实现

架构中的方法， 即**wrapper层**需要提供给应用层的功能， **driver层**需要操作屏幕的方法。 这些方法的**实现**放在了 posting层里。

在wrapper层不允许出现任何  与屏幕相关的信息出现。 同样， 在driver层， 不允许出现任何和芯片相关的操作函数， driver层，只需要关注如何实现操作外设。至于**操作外设**所需要的工具，由posting层提供。



**例**：在st7789.c中， 不能出现stm32f4的相关函数， 结构体等。 

简单来说， 在st7789的驱动文件中， 不能出现例如

```c
#include "stm32f4xx_hal.h"
```

这种， 需要包含芯片的头文件

## 对于触摸屏这种有特殊数据结构的解耦

对触摸屏这种具有特殊数据结构的解耦方式

```c
// 触摸数据结构体
typedef struct {
    uint8_t  touched; // 0-未按下，1-按下
    uint16_t x;
    uint16_t y;
} FT6336_Data_t;
```

应用层程序不可能直接定义一个 含有特定芯片名称的结构体， 所以需要特殊方式实现解耦：

对于驱动层， 不需要改变， 继续使用带芯片名称的结构体来操作， 我们需要在wrapper层下手， 定义一个

```c
typedef struct {
    uint8_t  is_pressed; // 0-抬起，1-按下
    uint16_t x;          // 通用X坐标
    uint16_t y;          // 通用Y坐标
} Touch_Data_t;
```

与具体芯片无关的结构体， 只与具体功能有关。

然后在posting层实现FT6336_Data_t --->Touch_Data_t的传递。



# MVC架构与消息泵模型



#### 1. MVC 架构对应关系：

- **Model（模型层 / 数据层）**：
    负责提供原始数据。也就是你的 **Service 层**（service_network.c，获取并解析天气和时间）。它根本不知道屏幕长什么样。
- **View（视图层 / 渲染层）**：
    负责把画面画出来。也就是那个一直跑 lv_task_handler 的 **LVGL 引擎任务**。它就像个瞎子画师，里面有什么数据它就画什么，它不负责逻辑。
- **Controller（控制层 / 业务总管）**：
    负责把 Model 的数据翻译给 View 听。也就是你新创建的那个 **UI 业务任务 (Task_UI_Logic)**。它死等数据，拿到数据后，拿着锁去修改 UI 控件。

#### 2. 消息泵模型 (Message Pump)：

设计的那个“专门接收 Service 发来数据的**消息队列 (Queue)**”，就是所谓的“消息泵”。
它就像一个邮局。Service 不断往里扔信（事件），Controller 从里面拿信。
Windows 系统的核心底层（GetMessage() 和 DispatchMessage()）就是完全一模一样的机制！
